# V2Next 当前 UI/功能问题修复与验证标准

Date: 2026-05-08

本文是本轮问题修复的执行契约。目标不是“把静态选项补齐”或“按截图堆样式”，而是恢复真实功能、统一页面结构，并删除不明所以的假入口/假设置。

## 0. 总原则

1. **先判定控件角色，再改实现**：列表行、页面入口、开关、菜单项、输入框、操作按钮、设置项不能混用。
2. **能用现有共享组件就用共享组件**：优先使用 `SecondaryListScaffold`、`PullRefreshListScaffold`、`ConciseListRow`、`GroupedListSection`、`AppTextField`、`AppActionButton`、`FilterChip` 等现有封装；不要在页面里继续散落 raw `Button`/`TextInput`/伪按钮。
3. **假功能必须消失或变成真功能**：没有路由、没有状态、没有持久化、没有点击反馈的选项不能继续作为“设置项”展示。
4. **副标题只保留有信息量的内容**：不要写“用于……”“点击进入……”这类自解释文案。副标题只用于当前状态、风险、限制或必要说明。
5. **修复必须闭环**：构建通过只是最低线；涉及 UI/交互的改动必须设备验证并截图/记录关键状态。

## 1. 问题清单与验收标准

### A. 数据/功能正确性 P0

- 本地内容项目后面的数字全部是 0。
  - 根因方向：设置页展示的统计来源/异步读取/初始化时机错误，而不是单纯文案问题。
  - 修复标准：数字来自真实本地存储统计；无数据时才显示 0；进入设置页、产生/删除本地内容后重新进入，统计会刷新。

- 节点页全是 10 个主题且不能加载更多。
  - 根因方向：节点主题列表分页/加载更多状态、下一页解析或请求参数错误。
  - 修复标准：首屏显示真实第一页；滚动到底部能请求下一页；没有更多时显示正确结束状态；错误时可重试。

- 主题页回复按钮打不开回复界面。
  - 根因方向：按钮绑定、登录状态判断、路由参数、ReplyEditor 路由注册或 sheet/modal 入口失效。
  - 修复标准：登录用户点击回复进入可编辑回复界面；未登录时给出登录路径或禁用解释；返回行为正确；不出现“点了没反应”。

### B. 页面结构和导航 P1

- 最近浏览、稍后读等列表页风格独立，没有使用现有列表页样式。
  - 修复标准：这些二级列表页使用现有列表 scaffold/row 体系；空态、加载态、分隔线、padding、安全区一致。

- 节点列表页的节点名和副标题信息应该放到 appbar 中。
  - 修复标准：节点页 appbar 显示节点名和必要副标题/统计；列表区域只负责主题内容，不重复大标题块。

- 用户页关注、屏蔽按钮应该放到 appbar menu 中。
  - 修复标准：关注/屏蔽属于页面级次要动作，放入 appbar menu；页面主体不再出现突兀的大按钮；登录/权限状态清晰。

### C. 设置页 IA 与组件滥用 P1

- 设置里的账号管理选项点不开。
  - 修复标准：如果已有账号管理页面/动作，则接真实路由；如果没有，就删除该入口或合并到当前账号区域，不能保留假入口。

- 阅读显示设置格格不入，行距/字号修改后同一行前面的数字不变，只有副标题变化。
  - 根因方向：把“可调值”做成了按钮/静态文案，状态展示分裂。
  - 修复标准：字号、行距等使用合适的调节组件或清晰的当前值展示；同一行只有一个权威当前值；修改后行内值与预览/副标题同步。

- 回复排序、只看楼主、代码换行三行意义不明且无设置/路由跳转功能。
  - 修复标准：若是可持久化偏好，改成真实开关/选择器并被主题页消费；若暂未实现，删除；不要保留无效静态项。

- 底部节点名输入框不明所以且高度离谱。
  - 修复标准：明确其角色。如果是节点跳转/搜索，改用共享输入框并提供明确 placeholder、提交行为和合理高度；如果没有真实用途，删除。

- 图片主机识别、外链打开等选项不明所以，既不像开关也不像页面入口。
  - 修复标准：按真实语义改为开关/选择器/入口；没有真实消费方则删除；不要以静态选项证明工程量。

- 域名切换滥用按钮设计。
  - 修复标准：域名/线路属于单选设置，使用列表单选/选择器/菜单，不使用一排按钮堆砌。

- 很多选项带有毫无必要的自解释副标题。
  - 修复标准：批量删除无效副标题，只保留状态类副标题，例如当前值、登录状态、缓存大小、风险提示。

## 2. 根因分类

1. **静态表驱动失控**：设置项先被写成静态数组，再补 UI，缺少真实路由/状态/消费方校验。
2. **组件角色混淆**：按钮被当作入口、选择器、开关和值展示；输入框被当作装饰块；副标题被当作解释文档。
3. **页面 scaffold 不统一**：二级列表页各自实现空态、padding、标题、分隔线，导致风格漂移。
4. **功能链路未验证**：回复、分页、统计等交互没有端到端设备验证，只做了静态布局或构建验证。
5. **AppBar 承载规则缺失**：页面标题、页面级菜单动作、主体内容边界不清。

## 3. 修复优先级与分批

### Batch 1：功能阻塞优先（P0）

范围：本地统计、节点分页、主题回复入口。

验收：
- `bash dev.sh --build-only` 通过。
- 设备安装后验证三条真实链路。
- 对失败/未登录/无更多数据状态有可见反馈。
- commit scope 建议：`fix(data)`、`fix(node)`、`fix(reply)` 或一个明确的 `fix(core)`，但必须说明 why/what/validation。

### Batch 2：列表页与 AppBar 结构（P1）

范围：最近浏览/稍后读/本地内容列表样式统一；节点页 appbar；用户页 appbar menu。

验收：
- 相关二级列表使用共享 scaffold/row。
- 节点名/副标题在 appbar 中，主体不重复。
- 用户关注/屏蔽在 appbar menu 中。
- 设备截图覆盖至少：列表页空态/有数据态、节点页、用户页 menu 打开态。

### Batch 3：设置页信息架构与控件替换（P1）

范围：账号管理、阅读显示、回复排序/只看楼主/代码换行、节点输入框、媒体/外链设置、域名切换、副标题清理。

验收：
- 每个设置项必须能回答：这是开关、选择器、页面入口、信息展示，还是删除？
- 无真实功能的入口删除，不用 disabled 假装完成。
- 字号/行距等当前值单一且同步。
- 域名切换使用单选/选择器；不再用按钮阵列。
- 设置页截图覆盖：主设置页、阅读显示区域、域名切换区域、媒体/外链相关区域。

## 4. 修改前检查清单

每改一个页面前，先记录：

- 页面文件与共享组件文件。
- 当前控件角色。
- 是否已有 HDS 或 shared wrapper 可用。
- 该选项/按钮/入口是否有真实路由、状态、持久化和消费方。
- 是否需要删除而不是修补。

## 5. 验证标准

### 必跑命令

```bash
bash dev.sh --build-only
git diff --check
```

HarmonyOS SDK 环境如需显式设置：

```bash
export DEVECO_SDK_HOME=/home/gamer/devtool/ohos/command-line-tools/sdk
```

### 设备验证

默认设备：`192.168.50.237:12345`。涉及 UI/交互时必须安装并验证：

- 相关页面能打开。
- 点击项有真实反馈或真实跳转。
- 返回键/返回手势正确。
- 列表滚动、加载更多、空态、错误态正确。
- 截图或 layout dump 记录关键状态。

### 提交要求

遵守 `AGENTS.md`：英文 Conventional Commit，包含 Why / What changed / Validation。禁止中文 commit message。

## 6. 禁止验收方式

- 只说“已统一样式”，但没有指出使用了哪个 shared component。
- 保留无法点击的设置入口。
- 把开关/选择器继续做成按钮。
- 用副标题解释显而易见的标题。
- 只构建不跑设备验证就提交 UI/交互改动。
